# План 02: Database and Auth

## Цель этапа

Поднять прикладной data layer и авторизацию на основе `Supabase Auth` и `Prisma`.

## Что входит

- проектирование и реализация начальной Prisma-схемы
- миграции для базовых таблиц
- интеграция `Supabase Auth`
- серверная модель сессии
- базовые guards для пользовательской части и админки

## Что не входит

- полный UI персонажей
- мастер жалобы в ОГП
- AI-сценарии

## Основные задачи

1. Создать таблицы `profiles`, `servers`, `user_server_state`, `characters`, `character_roles`, `character_access_flags`, `trustors`, `documents`, `ai_requests`.
2. Настроить связи с `Supabase Auth`.
3. Определить `DATABASE_URL` и `DIRECT_URL`.
4. Подготовить seed серверов.
5. Настроить login/logout и получение текущего пользователя.
6. Зафиксировать модель `super_admin` на уровне аккаунта.
7. Зафиксировать machine keys для `access_flags`, `user_access_flags` и `document_types` на уровне схемы и сидов.

## Критические правила этапа

- Prisma — основной прикладной слой работы с БД
- user identity приходит из `Supabase Auth`
- `super_admin` не зависит от персонажа
- состояние пользователя по серверу хранится в `user_server_state`
- soft delete должен быть учтен сразу в целевых таблицах

## Критерии завершения

- можно войти в приложение
- можно получить текущего пользователя на сервере
- Prisma-миграции выполняются стабильно
- базовые справочники и профили создаются корректно
