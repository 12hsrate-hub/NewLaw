# 20 — Навигация, рабочие зоны и правила входа в документы

## Назначение

Этот план фиксирует post-MVP линию по рефакторингу:

- глобальной навигации
- главной страницы
- пользовательских рабочих зон
- правил входа в document flows
- capability-based UX для доступов

План не меняет текущий runtime сам по себе.
Это source-of-truth для следующей продуктовой итерации по структуре интерфейса.

## Статус линии

- статус: `post-MVP / active planning line`
- тип: `product architecture + navigation refactor`
- dependency: использует уже существующие зоны `/account`, `/assistant`, `/servers`, `/internal`

## Что должно быть переосмыслено

### 1. Главная страница

Главная страница должна быть:

- панелью инструментов
- обзорной точкой входа
- быстрым переключателем между рабочими зонами

Главная страница не должна трактоваться как:

- центр документов
- расширенный `account`
- экран, требующий обязательного персонажа

### 2. Глобальный серверный контекст

Сервер должен быть глобальным переключаемым контекстом:

- доступным с любой пользовательской страницы
- сохраняемым как текущий рабочий сервер
- используемым как основной контекст для server-scoped модулей

При этом:

- сервер не должен превращать все страницы в document pages
- отсутствие выбранного персонажа не должно блокировать весь продукт

### 3. Персонаж как локальный, а не абсолютный контекст

Персонаж не является обязательным глобальным контекстом всего сайта.

Это означает:

- часть модулей работает без персонажа
- персонаж нужен только там, где он реально участвует в юридическом действии
- UI не должен требовать “сначала выбери персонажа” на всех экранах подряд

### 4. Ассистент как отдельный модуль

`/assistant` остаётся самостоятельным модулем.

Правила:

- ассистент работает без выбранного персонажа
- ассистент использует серверный контекст
- персонаж может быть optional context для формулировки запроса, но не hard requirement

### 5. Документы как отдельный модуль

Документы — это отдельный модуль, а не часть главной и не часть `account`.

Это означает:

- document entry должен жить в отдельной product zone
- `/account/documents` остаётся агрегатором и точкой обзора
- основной рабочий вход в документы должен идти через server-scoped documents module

### 6. Адвокатский кабинет как отдельный модуль

Нужен отдельный продуктовый модуль “Адвокатский кабинет”.

Этот модуль:

- не равен `account`
- не равен общему documents hub
- собирает только те возможности, которые требуют активного персонажа-адвоката

### 7. Account как зона настроек

`/account` должен трактоваться только как зона:

- настроек аккаунта
- безопасности
- управления доступами
- личных регистров и обзорных списков

`/account` не должен быть главным рабочим местом для документов и не должен подменять lawyer workspace.

### 8. Internal как скрытая служебная зона

`/internal` остаётся:

- скрытой служебной зоной
- не обычной пользовательской навигацией
- не частью публичной продуктовой карты

### 9. Capabilities вместо прямой role-check логики в UI

Пользовательский UI должен строиться от capabilities, а не от прямых проверок ролей в шаблонах.

Это означает:

- экран видит доступные возможности пользователя
- экран показывает только релевантные действия
- UI не строит продуктовые сценарии из прямого сравнения `roleKeys`

### 10. Правила выбора document family

Экран создания документа должен показывать только те типы документов, которые реально доступны пользователю в текущем контексте.

Нельзя:

- показывать все document families всем пользователям одинаково
- заставлять пользователя сначала выбирать искусственный режим работы

### 11. Запрет на свободный выбор режима

Нельзя проектировать свободный выбор режима:

- `разовый`
- `от себя`
- `для доверителя`

Режим должен определяться:

- выбранным типом документа
- capability пользователя
- наличием или отсутствием активного персонажа
- наличием или отсутствием доверителя в конкретном flow

### 12. Правила для жалобы от себя

Жалоба от себя требует:

- авторизации
- выбранного сервера
- активного персонажа

Без этого ordinary user не должен попадать в полноценный create flow.

### 13. Правила для адвокатских документов

Адвокатские документы требуют:

- активного персонажа-адвоката
- server context
- capability на соответствующий документ

### 14. Правила для договора

При создании `legal_services_agreement`:

- можно выбрать существующего доверителя
- можно создать нового доверителя в процессе
- доверитель должен сохраняться всегда
- доверитель должен быть привязан к конкретному персонажу-адвокату

### 15. Mobile scope

В рамках этого плана мобильная версия не проектируется.

Это означает:

- можно учитывать desktop-first структуру
- не нужно в этом этапе формализовать отдельную mobile IA

## UI baseline в рамках Плана 20

План 20 не является visual redesign line.

Но для новых и затронутых экранов в рамках этой линии нужен минимальный UI baseline, чтобы интерфейс не выглядел сырым.

Разрешено:

- нормальные карточки
- понятные заголовки
- аккуратные empty states
- единый стиль кнопок
- нормальные отступы
- user-friendly тексты на русском
- понятная шапка
- визуально чистая главная

Запрещено:

- полный visual redesign
- новая дизайн-система
- массовая смена цветов или темы
- сложные анимации
- редизайн всех старых страниц
- mobile-specific IA или mobile-first redesign
- переписывание editor forms только ради красоты
- изменение бизнес-логики ради UI

UI baseline первой волны применяется к:

- новой главной / product dashboard
- shared primary shell/header
- documents workspace entry
- lawyer workspace foundation
- access denied states
- empty states
- cards for available tools/actions

UI baseline первой волны не применяется к:

- persisted document editors
- `BBCode` preview/generation internals
- AI legal core
- AI review
- internal admin diagnostics
- old compatibility routes, если они не затрагиваются напрямую

Acceptance criteria:

- новые экраны используют существующие UI primitives/components, если они уже есть в проекте
- тексты на русском и понятны обычному пользователю
- недоступное действие объясняет причину и следующий шаг
- главная визуально читается как панель инструментов
- кнопки и карточки выглядят единообразно в пределах новых экранов
- в ordinary UI нет технических role/access/internal терминов
- UI changes не меняют capabilities и runtime authorization

## Предлагаемые slices

### 20.1 — Navigation and site map

- новая продуктовая карта зон
- новая роль главной страницы
- новый глобальный server switch

### 20.2 — Access capabilities contract for UI

- capability vocabulary
- mapping capabilities -> visible actions
- запрет на прямой role-first UX

### 20.3 — Document entry refactor

- create-document entry по capabilities
- lawyer-only document families
- запрет на свободный mode switch

### 20.4 — Lawyer workspace

- отдельный адвокатский модуль
- отделение от `/account`
- отделение от general documents hub

### 20.5 — Account cleanup after navigation split

- account как settings/security/access zone
- без притворства, что account — это главный рабочий экран продукта

## Критерии завершения линии

Линию можно считать закрытой, когда:

- продуктовая карта зон будет явно зафиксирована
- главная страница перестанет быть смешанной зоной
- server context станет глобальным и понятным
- assistant будет работать без персонажа как нормальный product flow
- documents и lawyer workspace будут отделены друг от друга
- account будет сужен до profile/settings/security/access scope
- UI document entry будет capability-based
- free mode switch `разовый / от себя / для доверителя` не появится
