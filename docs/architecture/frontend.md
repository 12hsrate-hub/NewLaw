# Frontend

## Назначение

Этот документ фиксирует базовый UI-стек MVP и правила границы между UI-слоем и доменной логикой.

## Базовый UI-стек MVP

Для MVP зафиксирован следующий UI-стек:

- `Tailwind CSS`
- `shadcn/ui`

Это базовый набор для первой версии.
В будущем допускается замена или эволюция UI-слоя, но без переписывания доменной логики и правил предметной области.

## Основные правила фронтенд-слоя

- UI-слой должен быть отделен от бизнес-логики.
- Базовые UI-компоненты должны жить отдельно от продуктовых компонентов.
- Продуктовые компоненты не должны напрямую содержать доступ к Prisma.
- Серверная логика и клиентская логика не должны смешиваться без необходимости.

## Базовые и продуктовые компоненты

Под базовыми UI-компонентами понимаются:

- переиспользуемые элементы интерфейса
- адаптированные компоненты на основе `shadcn/ui`
- общие layout-элементы

Под продуктовыми компонентами понимаются:

- компоненты конкретных сценариев Lawyer5RP
- компоненты wizard-шагов
- экранные блоки, привязанные к документам, персонажам и доверителям

Эти два уровня не должны смешиваться в один слой.

## Document wizard

- мастер документа не должен быть одним большим компонентом
- каждый шаг мастера должен быть отдельным компонентом или модулем
- UI шага должен быть отделен от серверных действий, валидации и генерации

## Граница устойчивости архитектуры

Доменные правила, работа со слепками, генерация BBCode, доступ к БД и AI-интеграция должны оставаться независимыми от конкретного UI-стека.
Это нужно для того, чтобы позже можно было менять внешний вид и UI-инструменты без переписывания предметной логики.
