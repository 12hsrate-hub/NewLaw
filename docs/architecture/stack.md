# Технологический стек

## Основной принцип

Для MVP выбирается один full-stack веб-проект без разбиения на monorepo и без отдельного backend-сервиса.
Это снижает организационную сложность, ускоряет старт и упрощает деплой на VPS.

## Зафиксированный стек

| Слой | Выбор | Роль в проекте |
| --- | --- | --- |
| UI и full-stack каркас | `Next.js` | единое приложение, маршрутизация, серверный рендеринг, server-side логика |
| UI-стилизация | `Tailwind CSS` | базовый utility-first слой стилизации MVP |
| UI-компоненты | `shadcn/ui` | базовый набор переиспользуемых компонентов MVP |
| Язык | `TypeScript` | типизация клиентской и серверной части |
| Роутинг | `App Router` | основной современный роутер `Next.js`, server components, route handlers, server actions |
| Валидация | `Zod` | единый слой схем и валидации |
| ORM и миграции | `Prisma` | схема БД, миграции, типобезопасный доступ к данным |
| База данных | `PostgreSQL` | основное прикладное хранилище |
| DB/Auth/Storage провайдер | `Supabase` | `PostgreSQL`, `Auth`, `Storage` в одной платформе |
| Аутентификация | `Supabase Auth` | login и identity-слой |
| AI | `OpenAI API` | серверная генерация и улучшение текста |
| Пакетный менеджер | `pnpm` | единый менеджер пакетов и lockfile |
| Production-хостинг | `Ubuntu 24.04 LTS VPS` | развертывание приложения и reverse proxy |

## Правила использования стека

### Next.js

- один репозиторий
- одно приложение
- отдельный backend-сервис не создается
- админка живет в том же приложении, а не в отдельном проекте
- `Server Actions` используются для внутренних действий UI
- `Route Handlers` используются для технических endpoint’ов

### Prisma

- Prisma является основным прикладным слоем доступа к БД
- миграции ведутся через Prisma
- схема БД хранится в репозитории и является частью version control

### Supabase

- `Supabase Auth` отвечает за identity
- `PostgreSQL` от `Supabase` является основной БД
- `Supabase Storage` остается частью платформы, но в document flow MVP не используется, потому что MVP работает со ссылками
- на этапе bootstrap не требуется отдельное техническое подключение `Storage`, если нет пользовательского сценария его использования

### UI-слой

- базовый UI-стек MVP: `Tailwind CSS` и `shadcn/ui`
- допускается будущая эволюция UI-слоя без переписывания доменной логики
- UI-правила детализированы в [frontend.md](./frontend.md)

### Zod

- `Zod` является единым слоем валидации
- схемы используются как основа для форм, серверных действий и технических endpoint’ов, где это применимо

### OpenAI API

- все вызовы идут только с сервера
- ключи не попадают в клиентский код
- все AI-запросы логируются в таблицу `ai_requests`
- единственный AI-сценарий MVP — улучшение описания для адвоката
- шаблоны AI prompt’ов хранятся в коде и репозитории, а не в БД

## Почему именно такой стек

- `Next.js` закрывает клиентскую и серверную часть без отдельного backend-проекта
- `App Router` подходит под современные серверные сценарии и встроенную админку
- `Supabase` сокращает инфраструктурный overhead на старте
- `Prisma` дает понятную схему и миграции для `PostgreSQL`
- `pnpm` хорошо подходит для одной кодовой базы без monorepo

## Окружения

Поддерживаются три окружения:

- `local`
- `staging`
- `production`

Для каждого окружения должны быть раздельные:

- `env`-переменные
- строка подключения к БД
- настройки Supabase
- настройки OpenAI
- домены и deployment-конфигурация

`staging` для MVP описывается так:

- тот же VPS
- отдельный процесс или сервис приложения
- отдельный домен или поддомен
- отдельные `env`
- отдельный `Supabase` project или отдельная staging БД
