# Доменная модель

## Базовая иерархия

Ключевая модель проекта:

`Account -> Server -> Characters -> Documents`

Это не просто визуальная иерархия, а правило проектирования.
Все основные пользовательские сценарии должны опираться именно на эту цепочку.

## Сущности

### Account

Аккаунт — это авторизованный пользователь системы.

На уровне аккаунта живут:

- связь с `Supabase Auth`
- глобальные пользовательские настройки
- признак `super_admin`
- account-scoped integration credentials и session foundation для будущей OGP forum automation

Зафиксированный `user_access_flag` уровня аккаунта:

- `super_admin`

`super_admin` не зависит от персонажа и не переносится между персонажами.
Это уровень аккаунта, а не уровень игрового профиля.

### Server

Сервер — это обязательный контекст, внутри которого живут персонажи, доверители и документы.

Правила:

- сервер выбирается заранее
- главный документный сценарий всегда запускается внутри выбранного сервера
- ограничения на персонажей и доверителей считаются внутри связки `account + server`
- серверное состояние пользователя хранится в `user_server_state` в БД

Отдельная пометка по law corpus и assistant:

- после блока law corpus появился отдельный public `server legal assistant`
- этот assistant является отдельным consumer подтвержденного law corpus сервера
- он не живёт внутри `/app` и не зависит от active server из личного кабинета
- он не смешивается с жалобами в ОГП, исками и пост-MVP template documents module
- judicial precedents после law corpus планируются как отдельный corpus, а не как разновидность `primary laws`
- до отдельного precedent workflow они не должны автоматически попадать в уже работающий assistant

Отдельная архитектурная note по размещению модулей:

- `account zone` — это отдельная пользовательская зона `/account`
- server-scoped функциональные модули должны жить отдельно от кабинета и брать `serverSlug` из URL, а не из last-used state
- internal `super_admin` и platform tools должны жить отдельно от пользовательского кабинета в namespace вида `/internal/...`
- текущие реализованные маршруты внутри `/app` нужно считать transitional состоянием, а не долгосрочной целевой архитектурой для всех новых модулей подряд

### Character

Персонаж — это основной автор документа внутри выбранного сервера.

Правила:

- на одном сервере максимум `3` персонажа на пользователя
- в рамках одного аккаунта и одного сервера нельзя иметь двух персонажей с одинаковым паспортом
- внутриигровой ник равен ФИО
- роли и флаги доступа привязываются к `character_id`, а не к имени
- у одного персонажа может быть несколько флагов доступа
- если персонажей на сервере нет, показывается empty state с предложением создать персонажа
- активного персонажа нельзя переключать в режиме редактирования документа
- активного персонажа нельзя удалить, пока не выбран другой, кроме случая когда это последний персонаж на сервере

Отдельная note по future document flows:

- в server-scoped document modules сервер определяется только через URL
- персонаж может подставляться как last-used UX-default для этого сервера
- пользователь всегда должен явно видеть, какой сервер и какой персонаж выбраны
- после старта документа персонаж фиксируется в snapshot документа
- `/account/characters` нужно трактовать как profile-management зону, а не как основной рабочий центр document flow

### Character roles и access flags

Модель ролей и флагов в MVP строится вокруг `character_id`.

Это означает:

- право подачи как гражданин или как адвокат определяется через профиль персонажа
- любые доступы должны ссылаться на стабильный `character_id`
- сравнение по ФИО не допускается

Зафиксированные `access_flags` уровня персонажа и сервера:

- `advocate`
- `server_editor`
- `server_admin`
- `tester`

Зафиксированные `document_types` для MVP и ближайшего расширения модели:

- `ogp_complaint`
- `rehabilitation`
- `lawsuit`

### Trustor

Доверитель — это отдельная сущность, живущая в контексте `user + server`.

Правила:

- доверитель не привязан к конкретному персонажу
- доверителя можно создать с минимумом полей
- доверителя можно дополнять позже
- доверителя можно удалить, не ломая исторические документы, потому что документ хранит слепок

### Document

Документ — это самостоятельный объект со своим жизненным циклом, версиями и слепком.

Правила:

- документ относится к конкретному `character_id`
- документ живет в серверном контексте
- документ может иметь ссылку на исходную карточку доверителя, но внутри себя хранит собственный слепок
- автор документа read-only и берется из персонажа
- данные доверителя внутри документа редактируются как слепок

Отдельная пометка после MVP:

- после MVP планируется отдельный модуль сервер-специфичных шаблонных документов
- этот модуль использует ту же базовую цепочку `Server -> Character -> Document`
- он не смешивается с жалобами в ОГП и исками
- он логически относится к server-scoped document area, а не к account zone
- его собственная подробная спецификация живет отдельно и не меняет текущий MVP-сценарий жалобы в ОГП

## Слепки

Слепок — центральное правило модели документов.

Правила:

- слепок фиксируется при первом сохранении черновика
- после фиксации документ больше не должен автоматически перезаписываться изменениями из карточки персонажа или доверителя
- автоподтягивание данных персонажа в документ работает только до первой генерации
- изменения доверителя внутри документа не обновляют карточку автоматически
- допускается отдельное явное действие для сохранения изменений обратно в карточку доверителя

## Жизненный цикл документа

### Черновик

Черновик существует до и после генерации.
После первого сохранения у него уже должен быть зафиксирован слепок.

### Генерация

При генерации документ получает:

- `last_generated_bbcode`
- `generated_at`
- `generated_law_version`
- `generated_template_version`
- `generated_form_schema_version`

### Изменения после генерации

Любая правка после генерации означает:

- документ помечается как измененный после генерации
- синхронизация сайта и форума сбрасывается

### Пересборка

Кнопка `Пересобрать` никогда не перезаписывает текущий документ.
Она всегда создает новый черновик-копию.

## Жалоба в ОГП как основной тип документа

В MVP главным документным сценарием считается жалоба в ОГП.

Правила сценария:

- сервер выбирается заранее
- персонаж выбирается заранее
- гражданский может подавать только от себя
- адвокат может подавать от себя и в интересах другого
- если подача от себя, блок доверителя скрывается и данные берутся из персонажа
- если подача как представитель, доверителя можно выбрать из списка или заполнить внутри документа
- результатом является BBCode для форума

## Доказательства

Доказательства строятся на серверных типах доказательств.

Для каждого типа:

- может существовать текст по умолчанию
- внутри типа может быть несколько строк
- каждая строка содержит свой текст и свою ссылку
- пользователь может менять текст строки
- пользователь может удалять стандартные строки
- пользователь может добавлять новые строки

## Публикация

В документе разрешена одна ссылка на публикацию.

Правила:

- ссылка публикации должна принадлежать домену `https://forum.gta5rp.com/`
- публикационный статус для MVP — `Опубликован на форуме`
- для опубликованного документа есть ручная отметка синхронизации сайта и форума

Отдельная note по forum automation foundation:

- account-scoped forum session хранится отдельно от документа
- документ не хранит raw cookies/token форума
- claims не наследуют автоматически OGP publication capability

## Неполный профиль персонажа

Если профиль персонажа неполный:

- мастер не блокируется на входе
- пользователь сразу видит предупреждение
- финальная генерация BBCode блокируется до заполнения обязательных полей персонажа
