# Структура кода

## Назначение

Этот документ фиксирует правила разбиения кода внутри одного full-stack приложения `Lawyer5RP MVP`.
Его цель — не допустить роста случайных комбайнов из UI, бизнес-логики, доступа к БД, AI и валидации в одном месте.

## Основной принцип

Код нужно раскладывать по ответственности, а не по случайному удобству одного файла.

Нельзя делать:

- большой файл, в котором одновременно живут `page`, JSX, вызовы Prisma, вызовы OpenAI, валидация формы и бизнес-правила
- клиентский компонент, который напрямую ходит в Prisma
- общий мастер документа как один большой модуль на сотни строк без разбиения по шагам и ответственности

## Базовые слои

В проекте нужно разделять как минимум следующие зоны:

- `page` / `layout`
- `components`
- `server/actions`
- `route handlers`
- `schemas`
- `db/repositories`
- `utils`

## Правила по слоям

### page и layout

- отвечают за композицию экрана и маршрута
- не должны содержать тяжелую бизнес-логику
- не должны напрямую обращаться к Prisma

### components

- отвечают за UI и локальное поведение интерфейса
- не должны напрямую использовать Prisma
- не должны содержать серверные секреты и прямые вызовы OpenAI
- базовые UI-компоненты должны жить отдельно от продуктовых компонентов

### server/actions

- содержат серверные точки входа для действий пользователя
- координируют бизнес-операции
- могут вызывать репозитории, схемы и серверные AI-модули

### route handlers

- используются для технических endpoint’ов
- не должны становиться вторым параллельным слоем для обычных внутренних UI-мутирующих действий
- используют те же схемы, репозитории и серверные модули, что и остальная серверная часть

### schemas

- содержат схемы валидации и нормализации входных данных
- базируются на `Zod`
- не должны дублироваться в нескольких местах без необходимости
- должны быть единым источником правды для форм, где это возможно

### db/repositories

- инкапсулируют доступ к Prisma и БД
- не должны содержать JSX и UI-код
- не должны превращаться в место для случайной бизнес-логики, не связанной с хранением данных

### utils

- содержат небольшие общие вспомогательные функции
- не должны становиться свалкой разнородной логики

## Отдельные правила для document wizard

- мастер документа не должен быть одним большим файлом
- каждый шаг мастера должен быть отдельным компонентом или модулем
- общие правила шага, схемы и преобразования данных должны жить отдельно от JSX
- генерация, сохранение черновика, работа со слепком и публикационная логика не должны смешиваться в одном UI-файле

## Правила для клиентской и серверной логики

- не смешивать клиентскую и серверную логику без необходимости
- серверные операции должны оставаться на сервере
- Prisma используется только в серверной зоне приложения
- OpenAI вызывается только из серверных модулей

## Правила против дублирования

- не дублировать бизнес-правила в нескольких местах
- не дублировать валидацию в нескольких местах без явной причины
- если правило уже зафиксировано в `schemas` или серверном модуле, UI не должен заводить свою вторую версию того же правила

## Правила против лишних абстракций

- не делать абстракции заранее без реальной пользы
- не выделять отдельные сервисные слои только ради архитектурной красоты
- сначала нужен понятный, локально читаемый модуль; затем уже обобщение, если оно действительно повторяется

## Safe refactor and read-path resilience policy

- крупные файлы режутся только по ответственности, а не по случайному механическому split
- compatibility barrel-export обязателен, если переносится публичный export
- document family labels, route builders и related metadata должны постепенно уходить в registry-подход
- client editors не должны одновременно содержать весь state, field rendering, evidence rendering, AI controls и generation/publication panels
- document read-model должен быть устойчив к legacy/broken JSON
- refactor PR не должен менять Prisma schema
- refactor PR не должен добавлять migrations
- refactor PR не должен менять пользовательское поведение
