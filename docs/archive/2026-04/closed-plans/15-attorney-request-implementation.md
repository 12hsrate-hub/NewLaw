## Критические правки относительно прошлого обсуждения

### 1. Не вводить signer mode enum

**Не делать** отдельные режимы вида:

- `LAWYER`
- `DEPUTY_HEAD_OF_BAR`

Это неправильная модель.

Для шаблона титул подписанта берётся из **должности персонажа**.

То есть:

- `Адвокат`
- `Заместитель Главы Коллегии Адвокатов`
- `Глава Коллегии Адвокатов`
- и другие релевантные должности

берутся из поля должности персонажа, а не из искусственного enum режима документа.

### 2. Для шаблона нужны render-формы должности

Из одной должности нужно получить несколько отображений:

- английский текст для левого столбца
- русский текст для вводного абзаца
- русский текст для подписи слева внизу

Пример:

#### Если должность = `Адвокат`
- left column EN: `Lawyer`
- body RU: `Адвокат Штата Сан-Андреас`
- footer RU: `Адвокат`

#### Если должность = `Заместитель Главы Коллегии Адвокатов`
- left column EN: `Deputy Head of Bar Association`
- body RU: `Заместитель Главы Коллегии Адвокатов штата Сан-Андреас`
- footer RU: `Заместитель Главы Коллегии Адвокатов`

#### Если должность = `Глава Коллегии Адвокатов`
- left column EN: `Head of Bar Association`
- body RU: `Глава Коллегии Адвокатов штата Сан-Андреас`
- footer RU: `Глава Коллегии Адвокатов`

Реализация:

- источник истины — **должность персонажа**
- в коде сделать **preset/mapping** для известных должностей
- при первом сохранении черновика **разрешить должность в render-формы и сохранить snapshot этих render-строк в документ**
- если должность не поддерживается для этого шаблона, **черновик сохранять можно**, но **генерацию блокировать** с понятной ошибкой

### 3. В v1 не делать блок приложений

Для `Адвокатского запроса` в первом варианте:

- **не делать attachments block**
- **не печатать приложения**
- **не добавлять лишние поля для приложений**

### 4. В v1 не делать сложную маршрутизацию Section 3

В первом варианте не нужна отдельная UI-модель для произвольной маршрутизации пунктов.

Section 3 в v1 собирается по простой схеме:

- пункт 3 → выбранный адресат из preset-списка
- пункты 1 и 2 → `Сотрудник / нашивка`

После автосборки **Section 3 можно редактировать вручную**.

### 5. Подпись персонажа для template documents

Следующий reusable step поверх уже реализованного `attorney_request`:

- подпись хранится в profile-management зоне персонажа как image asset
- asset принадлежит `character`, а не `account`
- `attorney_request` использует активную подпись персонажа автоматически
- при первом сохранении или первой генерации документа фиксируется frozen `signatureSnapshot`
- отсутствие активной подписи не блокирует создание черновика, но блокирует финальную генерацию
- будущий договор на оказание юридических услуг должен reuse-ить тот же `signatureSnapshot` contract, а не читать live character asset

## Scope v1

Реализовать новый document type:

- `ATTORNEY_REQUEST`

С правилами:

- документ обязательно создаётся в контексте **сервера**
- документ обязательно создаётся в контексте **персонажа**
- документ обязательно создаётся в контексте **доверителя**
- после **первого сохранения черновика** нельзя менять:
  - сервер
  - персонажа
  - доверителя
- snapshot персонажа и доверителя фиксируется при **первом сохранении черновика**
- результат генерации:
  - preview
  - PDF
  - JPG
- документ должен помещаться **ровно в 1 страницу**
- если не помещается — генерация блокируется

## Поддержка периода через полночь

Обязательно поддержать кейс:

- дата = `2026-02-01`
- `Время с` = `23:40`
- `Время по` = `00:20`

Это **валидно**, а не ошибка.

Логика:

- если `timeTo >= timeFrom` → период в пределах одной даты
- если `timeTo < timeFrom` → период переходит на следующий день

Нужно считать:

- `crossesMidnight`
- `periodStartAt`
- `periodEndAt`
- `periodDisplayText`

Для текста:

### Обычный случай
`за 28.03.2026 с 18:30 по 20:00`

### Через полночь
`за период с 01.02.2026 23:40 по 02.02.2026 00:20`

Именно такую логику заложить в builder Section 1.

## Валидация

### Черновик

Разрешить сохранять черновик даже при неполном наборе данных, если это соответствует общей логике проекта.

### Генерация блокируется, если:

- не выбран сервер
- не выбран персонаж
- не выбран доверитель
- нет номера запроса

Сделать normalizer с тестами.

- `2112` → `BAR-2112`

### Валидация времени

Тест-кейсы минимум:

- `18:30 -> 20:00` → обычный период
- `23:40 -> 00:20` → `crossesMidnight = true`

## Acceptance criteria

Считать задачу выполненной, если:

1. В системе появился новый тип документа `ATTORNEY_REQUEST`.
2. Документ можно создать только в контексте сервера + персонажа + доверителя.
3. После первого сохранения нельзя сменить сервер / персонажа / доверителя.
4. Номер запроса корректно нормализуется.
5. Кейс `BAR-BAR-2112` нормализуется в `BAR-2112`.
6. Поддержан период через полночь.
7. Должность подписанта берётся из должности персонажа, а не из signer mode enum.
8. Для неподдерживаемой должности генерация блокируется с понятной ошибкой.
9. Section 1 предзаполняется 3 пунктами и редактируется.
10. Section 2 и 4 автоматические.
11. Section 3 предзаполняется и редактируется.
12. Документ доступен из карточки доверителя.
13. В карточке доверителя виден список адвокатских запросов.
14. Работают preview + PDF + JPG.
15. Документ, не помещающийся в 1 страницу, не генерируется.
16. Не реализованы лишние приложения / attachments.
17. Не введён искусственный enum signer mode.
