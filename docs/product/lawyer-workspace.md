# Адвокатский кабинет

## Назначение

Этот документ фиксирует отдельный продуктовый модуль “Адвокатский кабинет”.

Он нужен, чтобы:

- не смешивать адвокатские сценарии с обычным `account`
- не прятать lawyer-only возможности внутри общего documents hub
- дать понятную рабочую зону для персонажа-адвоката

## Базовый принцип

Адвокатский кабинет — отдельный модуль.

Он:

- не равен `account`
- не равен assistant
- не равен общему documents module

## Что входит в адвокатский кабинет

Внутри lawyer workspace должны жить только сценарии, где реально нужен активный персонаж-адвокат.

Примеры:

- адвокатский запрос
- договор на оказание юридических услуг
- trustor-related flows, связанные с адвокатской работой
- другие future lawyer-only actions

## Что не должно жить в адвокатском кабинете

Не нужно тащить туда:

- общие настройки аккаунта
- общую безопасность аккаунта
- обычный assistant как модуль
- все документы подряд без разделения по capabilities

## Зависимости кабинета

Адвокатский кабинет требует:

- выбранный сервер
- активный персонаж
- этот персонаж должен иметь адвокатскую capability

Если этих условий нет, пользователь не должен видеть fake-ready workspace.

Вместо этого UI должен объяснить:

- чего не хватает
- какой следующий шаг нужен

## Связь с документами

Lawyer workspace не заменяет documents module.

Разница:

- documents module — общий модуль document flows
- lawyer workspace — curated entry для lawyer-only действий и связанных сущностей

## Связь с trustors

Trustors в рамках адвокатского кабинета важны не как глобальный registry ради самого registry.

Они важны как часть lawyer workflows:

- выбор доверителя
- создание доверителя в процессе
- сохранение доверителя при создании договора

## Правило для договора

`legal_services_agreement` внутри lawyer workspace должен:

- позволять выбрать существующего доверителя
- позволять создать нового доверителя в процессе
- всегда сохранять доверителя
- связывать этого доверителя с конкретным персонажем-адвокатом

## Почему нельзя оставлять всё в `/account`

Если lawyer-only flows остаются просто частью `/account`, возникают проблемы:

- account перегружается рабочими сценариями
- настройки и безопасность смешиваются с производственной работой
- адвокатский сценарий не выглядит как отдельное рабочее пространство

Поэтому `account` и `lawyer workspace` должны быть разведены.

## Что не проектируется в этом этапе

- mobile IA для lawyer workspace
- новая DB model для trustors
- runtime implementation lawyer dashboard
