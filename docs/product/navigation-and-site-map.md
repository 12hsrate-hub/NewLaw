# Навигация и карта зон

## Назначение

Этот документ фиксирует целевую карту пользовательских зон `Lawyer5RP / NewLaw` для post-MVP этапа навигационного рефакторинга.

Он не описывает runtime-реализацию.
Он описывает, как продукт должен быть разложен по модулям и рабочим входам.

## Базовый принцип

Главная страница — это панель инструментов.

Она:

- показывает основные продуктовые направления
- позволяет быстро выбрать сервер
- помогает войти в нужный модуль

Она не является:

- заменой documents module
- расширенным `account`
- местом, где пользователь постоянно работает с формами документов

## Основные пользовательские зоны

### 1. Главная

Роль:

- dashboard / tool panel
- быстрый переход в ключевые модули
- обзор доступных направлений

На главной допустимы:

- карточки модулей
- server switch
- быстрые ссылки на последние действия
- краткие onboarding-подсказки

На главной не должно быть:

- полного списка всех документов как primary flow
- логики “выбери режим документа и начни прямо здесь”
- смешения account settings и рабочей навигации

### 2. Servers

`/servers` — это слой выбора и входа в server-scoped модули.

Роль:

- выбрать сервер
- увидеть доступные модули по серверу
- войти в assistant, documents или lawyer workspace внутри серверного контекста

### 3. Assistant

`/assistant` и `/assistant/[serverSlug]` — отдельный модуль юридического помощника.

Роль:

- задать вопрос по законодательству и судебной практике сервера
- получить ответ с правовой опорой

Правило:

- assistant не требует выбранного персонажа
- assistant использует сервер как обязательный контекст

### 4. Documents

Documents — отдельный продуктовый модуль.

Роль:

- создавать документы
- открывать черновики
- работать с server-scoped document families

Основной рабочий вход:

- `/servers/[serverSlug]/documents/...`

`/account/documents` при этом остаётся:

- обзорной зоной
- личным списком
- агрегатором

Но не главным editor center.

### 5. Lawyer workspace

Это отдельный модуль для адвокатских сценариев.

Роль:

- собрать lawyer-only действия в одном месте
- не смешивать их с общими account settings
- не прятать их внутри общего documents hub без явного entry point

### 6. Account

`/account` — только:

- настройки аккаунта
- безопасность
- доступы
- персонажи
- доверители
- обзор документов

`/account` не должен быть основной производственной зоной документов и не должен подменять lawyer workspace.

### 7. Internal

`/internal` — отдельная скрытая служебная зона.

Она:

- не должна быть обычной пользовательской навигацией
- не должна описываться как равноправный public module

## Глобальные контексты

### Сервер

Сервер — глобальный контекст продукта.

Это означает:

- сервер можно переключить с любой пользовательской страницы
- server context влияет на assistant, documents и lawyer workspace
- server context не требует обязательного выбора персонажа

### Персонаж

Персонаж не является обязательным глобальным контекстом всего сайта.

Это означает:

- assistant работает без персонажа
- часть страниц account работает без активного персонажа
- документные и адвокатские flows могут требовать персонажа, но только локально

## Рекомендуемая верхнеуровневая карта

### Публично-пользовательские зоны

- Главная
- Серверы
- Ассистент
- Документы
- Адвокатский кабинет
- Account

### Скрытые служебные зоны

- Internal

## UI baseline в рамках этой линии

Этот документ не задаёт полный visual redesign.

Но для новых и затронутых экранов в рамках навигационного рефакторинга нужен минимальный визуальный baseline:

- чистые карточки для модулей и действий
- понятные заголовки и подзаголовки
- аккуратные empty states
- единообразные кнопки и CTA
- нормальные отступы
- понятная верхняя шапка
- user-friendly тексты на русском

Этот baseline применяется к:

- главной как tool panel
- shared primary shell/header
- server-scoped tool cards
- documents workspace entry
- lawyer workspace foundation
- access denied и empty states

Этот baseline не означает:

- новую дизайн-систему
- полный редизайн существующего продукта
- mobile-first IA
- переписывание editor forms только ради внешнего вида

Минимальные признаки готовности для новых экранов:

- пользователь с первого взгляда понимает, где он находится
- видно, какие действия доступны
- недоступное действие объясняет причину и следующий шаг
- экран не выглядит как raw technical scaffold

## Что не нужно проектировать в этом этапе

- mobile-first отдельную карту зон
- новую internal IA
- новую runtime permission system
