# Доступы и capabilities

## Назначение

Этот документ фиксирует product policy:

- пользовательский UI строится через capabilities
- product actions показываются через возможности пользователя
- UI не должен напрямую проектироваться от role-check списков

## Базовый принцип

В UI мы думаем не вопросом:

- “какая у пользователя роль?”

А вопросом:

- “что пользователь может сделать в текущем контексте?”

## Почему это важно

Прямая role-first модель в UI создаёт проблемы:

- экраны знают слишком много о внутренней модели ролей
- одинаковые проверки размазываются по множеству компонентов
- сложно объяснить пользователю, почему действие доступно или недоступно
- добавление новых документных сценариев становится хрупким

Capability-based слой нужен, чтобы:

- показывать только релевантные действия
- держать product wording на уровне пользовательских возможностей
- уменьшить зависимость UI от внутренней role vocabulary

## Типы capability-контекста

Capability должна вычисляться с учётом:

- авторизации
- выбранного сервера
- наличия активного персонажа
- наличия active advocate character
- типа документа или модуля

## Примеры product-level capabilities

Примеры capabilities, которые нужны именно на уровне UX:

- можно пользоваться assistant
- можно создать жалобу от себя
- можно открыть documents module
- можно работать с адвокатскими документами
- можно создать договор
- можно выбрать доверителя из registry
- можно создать доверителя в процессе
- можно войти в lawyer workspace

Это product wording.
Он не обязан один в один совпадать с внутренними кодовыми именами флагов и ролей.

## Чего не должно быть в обычном UI

Нельзя строить пользовательские экраны от прямой логики вида:

- “если есть роль X — покажи всё”
- “если роль не `advocate`, спрячь половину продукта без объяснения”

Ordinary user должен видеть:

- доступное действие
- что ещё нужно, чтобы оно стало доступно
- какой следующий шаг

## Правила для document entry

Экран создания документа должен:

- показывать только доступные типы документов
- скрывать недоступные типы или показывать их как недоступные с понятным объяснением
- объяснять недоступность через capability language

Например:

- нужен выбранный сервер
- нужен активный персонаж
- нужен персонаж с адвокатским доступом

Но не:

- “у вас нет roleKey X”
- “требуется accessFlag Y” как пользовательский текст

## Связь с персонажами

Персонаж важен, но не везде.

Правила:

- assistant не требует персонажа
- обычные server-scoped входы могут не требовать персонажа
- document flows и lawyer workspace могут требовать персонажа локально
- advocate-only flows требуют активного персонажа-адвоката

## Связь с account

`/account` не должен решать через role-check, что вообще является product navigation.

`/account` должен:

- показывать настройки
- показывать статус доступов
- помогать пользователю понять, какие возможности у него уже есть

Но не притворяться основным product router для всех действий.

## Что считается хорошим capability UX

Хороший UX говорит:

- что доступно сейчас
- чего не хватает
- что нужно сделать дальше

Примеры:

- “Для создания жалобы нужен активный персонаж.”
- “Для адвокатских документов нужен персонаж с адвокатским доступом.”
- “Ассистент доступен без персонажа, но требует выбранный сервер.”

## Что считается плохим capability UX

Плохой UX говорит:

- “У вас нет роли `lawyer`”
- “Требуется `accessFlag: advocate`”
- “Недоступно из-за role mismatch”

Такие формулировки допустимы только в internal/admin/diagnostic layer, но не как основной пользовательский продуктовый текст.
