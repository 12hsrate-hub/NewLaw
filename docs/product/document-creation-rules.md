# Правила создания документов

## Назначение

Этот документ фиксирует product policy для входа в document creation flows.

Он отвечает на вопросы:

- какие документы можно показывать пользователю
- от чего зависит доступность create flow
- какие режимы нельзя проектировать как свободный выбор

## Общий принцип

Создание документа должно показывать только те типы документов, которые доступны пользователю в текущем контексте.

Контекст включает:

- авторизацию
- выбранный сервер
- наличие активного персонажа
- наличие персонажа-адвоката там, где это требуется

## Что нельзя делать

Нельзя строить общий create screen с искусственным ручным выбором:

- `разовый`
- `от себя`
- `для доверителя`

Причина:

- это перекладывает внутреннюю логику document flows на пользователя
- это создаёт ложную универсальность
- это размывает реальные правила доступа

Режим должен следовать из:

- типа документа
- capability пользователя
- текущего контекста

## Общие правила входа

### Для обычной жалобы от себя

Требуется:

- авторизация
- выбранный сервер
- активный персонаж

Без этого полноценный create flow жалобы от себя не должен открываться как будто всё готово.

### Для адвокатских документов

Требуется:

- выбранный сервер
- активный персонаж
- этот персонаж должен иметь адвокатскую capability

К адвокатским документам в рамках текущей линии относятся:

- `attorney_request`
- `legal_services_agreement`
- другие lawyer-only flows, если они будут добавляться позже

### Для assistant

Это не document creation flow.

Поэтому:

- assistant не должен наследовать document creation requirements
- assistant не требует персонажа

## Правила для `legal_services_agreement`

При создании договора:

- пользователь может выбрать существующего доверителя
- пользователь может создать нового доверителя в процессе
- доверитель сохраняется всегда
- доверитель привязан к конкретному персонажу-адвокату

Это означает:

- create flow договора не должен требовать заранее готовый registry entry как единственный путь
- но и не должен оставлять договор без сохранённого доверителя

## Правила для trustor ownership

Доверитель не является абстрактной глобальной сущностью.

В рамках этой линии фиксируется:

- доверитель связан с конкретным персонажем-адвокатом
- trustor selection и trustor creation должны идти в контексте этого адвоката
- договор не должен создавать “ничейного” доверителя

## Что должен показывать create screen

Хороший create screen:

- показывает доступные document families
- объясняет, что нужно для недоступных действий
- не просит пользователя сначала разбираться во внутренних режимах

Примеры:

- `Жалоба в ОГП`
- `Исковое заявление`
- `Адвокатский запрос`
- `Договор на оказание юридических услуг`

Если документ недоступен, лучше показывать:

- понятную причину
- следующий шаг

Например:

- “Для этого документа нужен активный персонаж.”
- “Для этого документа нужен персонаж с адвокатским доступом.”

## Что не нужно фиксировать в этом этапе

- low-level runtime checks
- schema changes
- migration trustor model
- mobile create UX
